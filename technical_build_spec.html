Storyboard AI — Technical Build Spec

Version: V1 engineering build spec
Status: Ready for implementation
Purpose: Translate the settled behavioral contract in the Product Brief V2 and Implementation Pack into an engineering blueprint a team can build from immediately.

This document intentionally does not restate product strategy or behavioral rules except by reference to implementation-pack IDs such as M-05, F-04, and AT-15.

0. Engineering Principles

  • V1 architecture: modular monolith, not microservices.
  • Commands mutate canonical state. Projections derive visible truth.
  • SSE signals change. GET /scenes/:id/workspace delivers truth.
  • Authoritative state changes are transactional; generated artifacts are asynchronous, snapshot-based, idempotent side effects.
  • Provider output is provisional until stored, validated, version-checked, and promoted.
  • A panel is a logical record. An image is a versioned asset. Promotion changes pointers, not files.

1. System Architecture

1.1 V1 deployment model

  • Web app: Next.js 15 App Router frontend.
  • API + domain backend: NestJS TypeScript app in the same repo and deploy unit.
  • Worker: separate process from the same codebase for async jobs triggered by M-05, M-12, and M-15.
  • Primary DB: PostgreSQL 16.
  • Queue / locks: Redis + BullMQ.
  • Blob storage: S3/R2-compatible object store.

1.2 Why modular monolith

OptionBuysCostsRecommendation
Modular monolithFastest iteration, shared types, easier tracing, lower infra complexity, coherent transactionsLess deploy isolation, larger codebase boundaries to enforce manuallyChoose for V1
MicroservicesIndependent scaling, harder isolationSchema churn, distributed tracing overhead, more failure modes, slower team velocityDefer until post-V1 scale pressure

1.3 Internal module boundaries

  1. workspace-api — command endpoints, workspace projection endpoint, SSE.
  2. ingest-parse — screenplay ingest, format detection, parse, confidence scoring.
  3. shot-planner — beat annotations, shot recommendations, rationale generation, linting.
  4. render-orchestrator — render intent creation, asset prep, queueing, promotion, validation.
  5. export — export snapshot creation, PDF generation, signed delivery.
  6. projection — derived-state engine and SceneWorkspace assembly.
  7. telemetry — event capture, outbox dispatch, monitoring hooks.
  8. auth-access — authentication, project authorization, signed asset policies.

1.4 State placement

  • Canonical state: Postgres.
  • Ephemeral state: Redis queue metadata, locks, rate-limit counters.
  • Artifacts: object store only.
  • Projection cache: optional materialized/cached scene workspace keyed by scene_id + scene_version; rebuildable from canonical rows.

1.5 Command/query separation

  • Command endpoints mutate narrow aggregates with optimistic concurrency and idempotency.
  • Queries read a server-projected SceneWorkspace DTO.
  • The client never derives stale, blocked, failed, or export-ready truth from raw rows.

1.6 Async architecture

  • Queues: render-high (for M-12), render-normal (for M-05), export.
  • Outbox: DB outbox table for projection refresh and telemetry dispatch after commit.
  • Sweeper jobs: recover committed mutations without queued jobs, completed jobs without projection refresh, stored assets without validation record.

2.1 Image generation provider strategy

  • Implement one render adapter interface with exactly two concrete providers: one primary, one fallback.
  • Candidate primary: provider with strongest multi-image conditioning and continuity adherence in benchmark.
  • Required adapter methods: buildPromptBundle(), buildReferenceSet(), submitRenderJob(), normalizeProviderResponse().
  • Run a provider evaluation gate before locking primary: 20-scene benchmark on shot adherence, continuity, retry drift, cost, latency.

3.6 Storage strategy for images

  • All binaries in object storage.
  • Immutable key shape: projects/{project_id}/scenes/{scene_id}/panels/{panel_id}/variants/{variant_id}/{asset_kind}.
  • Asset kinds: original, normalized, thumb_sm, thumb_md, export_pdf, optional debug_manifest.json.
  • Never overwrite in place.

3.7 Migrations

  • Use forward-only Prisma migrations.
  • Ship enums and core tables first.
  • Backfill nullable fields before adding not-null constraints.
  • Never make asset object keys part of business logic; IDs only.

4.7 SSE contract

  • SSE is notification only, never authoritative state transport.
  • Allowed uses: badges for queued/started, spinner presence, refresh trigger.
  • Not allowed: patching canonical freshness, promoting variants, clearing blockers.
event types:
- projection.updated
- job.queued
- job.started
- job.completed
- job.completed_stale
- job.failed
- validation.failed
- export.completed

required envelope:
- event_id
- scene_id
- scene_version
- projection_version
- event_type
- emitted_at
- job_id (if applicable)
- mutation_ref | job_ref
- affected_ids
- reason_codes
  • Client applies transient updates only if projection_version > current_projection_version.
  • If versions skip, regress, or reference unknown entities, client does full refetch.
  • After job.completed, job.completed_stale, validation.failed, export.completed: refetch by default.

5. Rendering Pipeline

5.1 Core rule

The worker receives a fully materialized, persisted, versioned RenderIntent / RenderInputSnapshot, never live workspace state.

5.2 RenderIntent fields

  • project_id, scene_id, beat_id, shot_id, panel_id
  • mutation_ref (M-05 or M-12)
  • style_profile_snapshot
  • shot_spec_snapshot
  • grounding_refs[]
  • continuity_context (prior panel refs + pinned anchor + continuity targets)
  • negative_constraints
  • seed_policy
  • output_tier
  • attempt_number
  • scene/shot/panel versions at enqueue
  • reference_bundle_id
  • render_intent_hash

5.3 Prompt construction

Each render request is assembled from four layers:

  1. Shot block — literal approved shot spec from M-05/M-12; renderer must not improvise shot grammar.
  2. World block — style profile snapshot plus grounding card traits and reference handles.
  3. Continuity block — preserve facts from anchor panel, prior panel, and structured continuity targets.
  4. Constraint block — negative prompt and do-not-change list.

Prompt payload shape:

  • System: cinematic storyboard renderer; obey shot grammar exactly.
  • Developer: preserve anchors, do not add new characters/props, match framing/angle/type exactly.
  • User: beat content, emotional intent, scene context.
  • Attachments: grounding card images, optional pinned anchor, optional prior accepted panel.

5.4 Continuity enforcement

Do not rely on prompt text or seeds alone. Use layered continuity:

  1. Reference image conditioning — canonical character/location/prop refs.
  2. Pinned anchor support from M-13 with higher-weight image reference.
  3. Structured continuity facts — wardrobe, handedness, prop possession, time of day, location features, camera side.
  4. Canonical descriptors — generated once and reused verbatim.
  5. Shot-local seed reuse only as retry stabilizer.
  6. Post-render validation — vision pass for mismatch/drift.

Entity continuity graph: internally model relationships among characters, props, locations, shots, panels, and anchors so M-13/M-14 invalidation and continuity selection stay deterministic instead of conditional spaghetti.

5.5 Local-edit guarantee for M-12

  • M-12 creates a new panel_variants row for the target panel only.
  • Neighbor panels are never re-queued implicitly.
  • Neighbor prompt bundles are never recomputed.
  • Neighbor state rows are never mutated during regen.
  • Allowed writes: target panel current variant pointer (after promotion), target continuity assessment, append-only variant history, telemetry.
  • Any downstream healing must be a separate explicit mutation, not hidden behavior.

5.6 Render job flow

  1. Command transaction writes mutation row, invalidation results, render snapshot, job stub.
  2. After commit, outbox or queue dispatch enqueues render job with idempotency key.
  3. Worker loads snapshot and asset bundle.
  4. Asset preflight normalizer prepares provider-safe references.
  5. Provider call executes.
  6. Returned artifact stored in object storage and recorded in assets.
  7. Validation runs.
  8. Version check runs against current scene/shot/panel versions.
  9. If validation and version match pass, transactional promotion updates current_variant_id.
  10. Projection refresh + telemetry emit.

5.7 Asset preflight normalizer

  • Fetch signed source asset
  • Validate MIME, dimensions, file size
  • Strip EXIF/metadata
  • Transcode to provider-safe format
  • Optionally crop/focus
  • Generate deterministic hash
  • Write normalized asset + lineage row

5.8 Validation

Two-stage validator:

  1. Cheap structural validator: corruption, dimensions, blank/low-detail detection, basic shot class check.
  2. Semantic validator: expected entities present, forbidden entities absent, shot adherence, continuity anchor preservation.

If stage 1 fails, skip stage 2. If validation fails, artifact remains stored but non-current and maps to F-04 or F-06.

5.9 Queue policy

  • Priority: M-12 local regen outranks M-05 scene render.
  • Concurrency: start 2–4 concurrent renders per provider budget lane.
  • Timeout: panel render 90s hard timeout.
  • Retries: provider/network errors: 2 with exponential backoff; deterministic validation mismatch: 1 constrained retry; repeated failure -> terminal job failure.
  • Idempotency: dedupe on render_intent_hash.

5.10 Promotion preconditions

  • asset row exists
  • object upload confirmed
  • hash verified
  • validation record exists and passed
  • completion version check passed

Otherwise status is non-promotable: FAILED_STORAGE, FAILED_VALIDATION, or COMPLETED_STALE.

5.11 Cost model

  • Parse: < $0.02 / scene
  • Shot plan: < $0.05 / scene
  • Panel rough render: $0.03–0.08
  • Panel presentation render: $0.08–0.25
  • 8-panel scene render envelope: roughly $0.64–2.00 before retry/export overhead

6. Parse Pipeline

6.1 Format detection and validation for M-01

  • Support Fountain first; accept plain text screenplay-ish input with heuristics.
  • Rule-based detection for sluglines, dialogue blocks, action blocks, parentheticals, transitions.
  • Validation failures map to F-01/F-02 with scene/line-level reason codes.

6.2 Hybrid parse approach

  1. Deterministic parse for structural segmentation.
  2. LLM normalization pass for malformed repairs, beat extraction, entity normalization.
  3. Confidence scoring pass combining structural certainty, model certainty, entity resolution certainty, and anomaly count.

6.3 Confidence methodology

scene_confidence = weighted(rule_confidence, llm_confidence, entity_resolution_confidence, anomaly_penalty)

  • Rule confidence: structural parser match quality.
  • LLM confidence: model self-report + schema completion quality.
  • Entity resolution confidence: normalized entity linkage certainty.
  • Anomaly penalty: malformed spans, conflicting scene boundaries, unresolved names.

6.4 Domain mapping

  • script text -> scripts + script_lines
  • scene boundaries -> scenes
  • beat spans -> beats
  • entity mentions -> entity_occurrences + grounding seed candidates
  • confidence and provenance stored at scene/beat/entity level

7. Shot Plan Generation

7.1 Two-stage pipeline for M-02

  1. Beat annotation stage: derive semantic features — subject, reveal/withhold, motion, emotional intensity, geography, POV relevance, continuity burden.
  2. Recommendation stage: structured LLM output produces ShotRecommendation[].

7.2 Output schema

  • shot_type
  • framing
  • angle
  • intent
  • rationale
  • continuity_note
  • source_beat_span_ids[]

7.3 Rationale generation

  • Rationale must cite concrete beat facts, not generic cinematography filler.
  • Reject and regenerate rationale if it lacks beat evidence.
  • Store source beat span references for auditability.

7.4 Genre preset influence

  • Genre preset acts as a weighting layer, not a template override.
  • It influences preferred framing frequency, motion tolerance, intimacy distance, and reveal pacing.

7.5 Anti-cliché guardrail

  • Negative prompt/rules: avoid defaulting to establishing + shot-reverse-shot unless beat structure justifies it.
  • Lint output for repeated shot types, boilerplate rationale phrases, monotony score.
  • If lint fails, regenerate before persisting rows.

8. Infrastructure, Performance, and Cost

8.1 Expected latency envelopes

Pipeline stageTarget latency
M-01 parse2–10s per scene depending on script length
M-02 shot plan2–8s per scene
M-05 panel render15–60s per panel typical, 90s hard timeout
M-12 local regensame as panel render but priority queue
M-15 export5–20s typical, 60s hard timeout

8.2 V1 scaling assumptions

  • Acceptable: low hundreds of active projects, dozens of concurrent render jobs, one-scene-focused usage pattern.
  • Likely breakpoints: provider rate limits, render spend spikes, large-project workspace hydration if projection caching is absent.
  • Hard V1 limits: max screenplay pages, max scenes/project, max reference images/grounding card, max concurrent renders/user, max regen burst/minute.

8.3 Monitoring and alerting

  • Track all command completions by mutation_ref.
  • Alert on provider error spikes, queue backlog, validation failure rate, stale-on-arrival rate, export failure rate, asset integrity failures.
  • Tie telemetry event names to M-xx, AT-xx, F-xx references.

9. Security and Data Handling

  • User isolation: every project scoped by workspace_owner_id.
  • Project access: V1 single-owner access model; no multi-user collaboration complexity yet.
  • Asset access: signed URLs only; immutable objects private by default.
  • Screenplay handling: store raw text encrypted at rest if provider or customer sensitivity requires; redact raw screenplay content from provider payload logs by default.
  • Provider minimization: send only required scene/beat context to model calls, not whole project if avoidable.
  • Retention: retain generated assets and snapshots while referenced; purge unreferenced temp normalized assets on retention schedule.

10. Build Sequence

  1. Schema backbone: core tables, enums, assets, jobs, idempotency, outbox.
  2. Projection read side: SceneWorkspace DTO + derivation engine + workspace endpoint.
  3. M-01 parse pipeline with confidence and provenance.
  4. M-02 shot plan generation with rationale linting.
  5. M-03/M-04/M-06/M-07/M-08/M-09/M-10/M-11 command endpoints with version checks.
  6. Worker scaffold: BullMQ, outbox, sweeper, SSE notification path.
  7. M-05 scene render with render snapshot persistence.
  8. M-12 local regen with append-only variant history and mutation isolation.
  9. M-13/M-14 continuity invalidation plumbing.
  10. M-15 export pipeline using export snapshots and PDFKit.
  11. Post-render validator for F-04 / F-06.
  12. Telemetry + alerting aligned to matrix IDs and acceptance tests.

10.1 What can be stubbed for vertical slice

  • Fallback image provider can be stubbed behind adapter.
  • Projection cache can be omitted initially if workspace endpoint remains single-query-assembly efficient.
  • Semantic validator can start simple if structural validator and manual failure handling exist.
  • Reference-asset cropping heuristics can be basic; normalization itself cannot be skipped.

10.2 Vertical slice complete when

  • One scene can move end to end through M-01 -> M-05 -> M-12 -> M-15.
  • AT-01 through AT-20 can execute against seeded data or automated fixtures.
  • M-12 provably writes no sibling panel mutations.
  • Every mutation emits telemetry with mutation_ref.
  • Worker completions refresh projections and SSE notifies clients.
  • Export artifact is generated from persisted snapshot state, not client memory.

11. Technical Risks to Record Explicitly

  • Provider lock risk: continuity quality may depend on provider-specific multi-image conditioning behavior; mitigate with adapter and benchmark gate.
  • UI state drift risk: if frontend infers behavioral truth, screen contract collapses; mitigate with server projection law.
  • Cost risk: margins fail fast if average regens/panel exceed assumptions; enforce rate limits and cost telemetry from day one.
  • Projection performance risk: scene hydration degrades without careful indexing or projection cache; benchmark early.

12. Non-Negotiable Spec Lines

  • The client never reconstructs canonical workspace state from event history.
  • Commands are versioned and idempotent. Projections are rebuildable.
  • Provider completion is not acceptance.
  • Asset existence, integrity, validation, and version-match are all required before a generated artifact can become current state.
  • Canonical state points to immutable assets by ID; storage objects are append-only, lineage-tracked, and never overwritten in place.
